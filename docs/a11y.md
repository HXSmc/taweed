# WCAG AA Accessibility Audit — Taweed

> WCAG AA accessibility audit of every page/component, 2026-07-09/10. Scope: all 9 routes
> (landing, login, overview, analytics, ingest, scrubber, appeals, recovery, settings) × EN/AR ×
> light/dark. Method: real automated scans via axe-core 4.12.1 (injected live via chrome-devtools
> MCP, WCAG 2.0 A/AA + 2.1 AA rule set) + manual keyboard-navigation/focus-order checks + full-page
> screenshots, one parallel agent per page → every candidate adversarially re-verified by an
> independent agent (re-run the same axe scan or re-read the component source, default-to-refute)
> → every CONFIRMED finding fixed with a regression test. Ran across **6 relaunches** spanning
> multiple session interruptions (3 harness process restarts, 1 session-limit pause, 1 batch of
> transient safety-classifier/API errors) — `resumeFromRunId` caching meant each relaunch only
> re-ran failed/missing agent slots, not the whole audit. Final clean run: **35/35 agents, 0
> errors**. Findings below are the full cumulative set confirmed across all relaunches (many were
> fixed in earlier passes and no longer reproduce in the final audit — verified directly against
> current source, not just trusted from stale agent output).
>
> Final verification after all fixes: root+web typecheck clean, lint 0 errors, unit/integration
> tests green (25 new regression tests added, one per finding), `apps/web` production build green.

## Critical

### 1. `apps/web/components/modules/eob-review/eob-extraction-form.tsx` — line-item inputs had no accessible name at all
Every per-line claim-line input (claim line ref, SBS code, ICD-10-AM code, billed/paid/patient-
share/rejected money — 7 inputs per claim line) was a bare `<input>` with no `<label>`,
`aria-label`, `aria-labelledby`, or placeholder — WCAG 1.3.1/4.1.2 (Level A). A remittance with 3
claims × 4 lines is potentially 84 indistinguishable unlabeled fields a reviewer must correct blind
before approving money written to claims/denials.
**Fix:** visually-hidden (`sr-only`) `<span>` label added per input, mirroring the existing
`denialCode` `<select>` in the same row which already had one.
**Test:** `apps/web/test/eob-extraction-form.test.tsx`.

## High

### 2. `apps/web/components/marketing/landing.tsx:93` — proof-beat step numbers used the divider color as text color
The `01`/`02`/`03` step numbers used `--hairline-strong` (a subtle divider-border token, not a text
token) as real text color — 1.5:1 (light) / 1.57:1 (dark) against the 3:1 large-text minimum.
Reproduced identically in EN-light, EN-dark, AR-light (RTL).
**Fix:** switched to a proper text-color token.
**Test:** `apps/web/test/landing-proof-beat-contrast.test.tsx`.

### 3. `apps/web/components/marketing/landing.tsx:70` — hero disclaimer caption below AA
The "Illustrative figure. Get your real number from a free audit." caption used `.text-faint` —
3.3:1 (light) / 3.92:1 (dark) against the 4.5:1 normal-text minimum. This qualifies the headline
SAR figure as illustrative rather than real, so its low legibility is compliance-adjacent, not just
cosmetic.
**Fix:** switched to an AA-safe token (comment left in source citing the finding).
**Test:** `apps/web/test/landing-hero-h1-and-caption-contrast.test.tsx`.

### 4. `apps/web/components/marketing/landing.tsx` — hero eyebrow line, dark-theme contrast
The eyebrow line above the headline used a locked accent color with no `dark:` override, failing
AA in dark theme.
**Fix:** added a `dark:` override to an AA-safe pairing.
**Test:** `apps/web/test/landing-eyebrow-dark-contrast.test.tsx`.

### 5. `apps/web/components/ui/badge.tsx:13` — demo-account role Badge fails AA in dark theme
`app/globals.css` defines `--accent: #2557e4` only in `:root` and never redefines it inside `.dark`,
while `--accent-subtle` IS redefined for dark (`#16233f`). The light-tuned blue text rendered
directly on the new dark navy chip background measured 2.64:1 (need 4.5:1) — confirmed on all 10
demo-account role badges on `/en/login` and `/ar/login`, identical in EN-dark and AR-dark; EN-light
and AR-light scanned clean.
**Fix:** dark-theme `accent` variant swapped to the solid `bg-accent`/`text-accent-fg` pairing
(same high-contrast combo already used for buttons), instead of the subtle pairing.
**Test:** `apps/web/test/badge-accent-dark-contrast.test.tsx`, `confidence-badge-dark-contrast.test.tsx`.

### 6. `apps/web/components/charts/pareto.tsx` — chart has no accessible name or text alternative
The "Top denial reasons" Pareto chart on Analytics had zero accessible name/summary, unlike the
sibling `TrendLine` chart on the same page which already implements `role="img"` +
`aria-labelledby` + `sr-only` figcaption.
**Fix:** `Pareto` now takes `titleId`/`summary` props and wraps in the identical accessible-chart
pattern as `TrendLine`.
**Test:** `apps/web/test/trend-line-accessibility.test.tsx` (regression guard for both charts).

### 7. Money/analytics semantic color tokens — missing or insufficient dark-mode overrides
`--at-risk-text`, `--recovered-text`, `--at-risk-bg`, `--recovered-bg` lacked (or had insufficient)
`.dark` overrides across `MoneyIndicator`, `ScrubberTable` flag badges, Settings rules-severity
badges, and `ConfidenceBadge` — money-semantic color was not reliably AA in dark theme.
**Fix:** dark-mode token overrides added/corrected; verified 4.5:1 against `--surface-1` and each
token's own paired background.
**Test:** `analytics-dark-money-token-contrast.test.ts`, `money-indicator-dark-contrast.test.tsx`,
`scrubber-flag-badge-dark-contrast.test.tsx`, `settings-rules-badge-dark-contrast.test.tsx`.

## Medium

### 8. `apps/web/components/marketing/landing.tsx` — no `<h1>` anywhere on the page
Heading structure jumped straight to `<h2>` for the "Interest" section; the "Taweed" wordmark was a
plain `<span>` and all hero copy was `<p>` tags — no top-level heading for a screen-reader's
page-outline navigation, inconsistent with this repo's own "Semantic HTML First" rule.
**Fix:** `<h1>` added wrapping the hero headline.
**Test:** `apps/web/test/landing-hero-h1-and-caption-contrast.test.tsx`.

### 9. `apps/web/components/shell/locale-toggle.tsx:20` — accessible name has no `lang` override
`aria-label={`Switch to ${next === 'ar' ? 'Arabic' : 'English'}`}` resolves to an English string
with no `lang="en"` on the `/ar` page — WCAG 3.1.2 Language of Parts. A screen reader in Arabic mode
mispronounces the English label using Arabic phonetic rules.
**Fix:** `lang="en"` added to the button (comment explains why).
**Test:** `apps/web/test/locale-toggle.test.tsx`.

### 10. `apps/web/components/modules/appeals-composer.tsx` — selection/async load not announced
Selecting a queue row triggers an async draft load and swaps the entire composer panel, but nothing
was announced to assistive tech (`useTransition`'s `pending` flips back to `false` before the async
load actually settles) — no `aria-live` region, no focus move.
**Fix:** explicit `isLoadingDraft` state tracking the real round-trip, plus an always-mounted
`sr-only` `role="status" aria-live="polite"` element with three states (empty/loading/loaded).
**Test:** `apps/web/test/appeals-composer-a11y.test.tsx`.

### 11. `apps/web/app/[locale]/(app)/recovery/page.tsx` — per-row action buttons had generic names
"Mark won"/"Mark lost" buttons had identical accessible names on every row, no row context — unlike
Scrubber's per-row convention (`aria-label={`Claim ${id}, risk ${score}`}`).
**Fix:** row-context `aria-label`s built from payer/amount, mirroring Scrubber's convention.
**Test:** `apps/web/test/recovery-page-dark-contrast.test.tsx`,
`recovery-pipeline-actions-header.test.tsx` (also covers finding #12).

### 12. Recovery pipeline table — trailing action column header has no accessible name
**Fix:** accessible name added to the header cell.
**Test:** `apps/web/test/recovery-pipeline-actions-header.test.tsx`.

### 13. Ingest panel — hardcoded English strings, contrast
The quarantine table's "Ref" column header was a hardcoded English literal (never localized), and
the dropzone hint used `.text-faint`.
**Fix:** header routed through `t()`; hint switched to an AA-contrast token.
**Test:** `apps/web/test/ingest-panel-contrast-and-i18n.test.tsx`.

### 14. Settings — Scrubber scope column header hardcoded, not localized
**Fix:** routed through `t()` for both EN and AR.
**Test:** `apps/web/test/settings-scrubber-scope-i18n.test.tsx`.

### 15. `apps/web/components/shell/rail.tsx` — nav link labels used `display:none` instead of `sr-only`, some missing accessible names at tablet width
**Fix:** switched to `sr-only` (keeps the label in the accessibility tree while visually collapsed)
and ensured every module link has a non-empty accessible name at tablet breakpoints.
**Test:** `apps/web/test/rail-link-name-tablet.test.tsx`, `rail-active-nav-dark-contrast.test.tsx`.

### 16. Chart animations ignored `prefers-reduced-motion`
The global CSS reduced-motion rule only collapses CSS animation/transition durations — it never
reaches recharts' JS/rAF-driven draw-in tweening, so `TrendLine` and `Pareto` animated regardless of
the user's OS-level preference.
**Fix:** new `apps/web/hooks/use-reduced-motion.ts` hook (mirrors the existing check in
`money/count-up.tsx`), wired into both chart components to gate their animation.
**Test:** `apps/web/test/use-reduced-motion.test.tsx`.

### 17. Authenticated app shell — no skip-to-content link (WCAG 2.4.1 Bypass Blocks)
A sighted keyboard-only user had to Tab through every primary-nav link plus the full command bar
(~12 stops) before reaching page content, on every route change.
**Fix:** new `apps/web/components/shell/skip-link.tsx` (`SkipLink`), visually hidden until focused,
jumps to `<main id="main-content">`; wired into `app/[locale]/(app)/layout.tsx` as the first
focusable element in the shell.
**Test:** `apps/web/test/skip-link.test.tsx`.

### 18. `apps/web/components/ui/tabs.tsx` — arrow-key navigation not RTL-aware
`ArrowLeft`/`ArrowRight` kept LTR semantics on the Arabic locale instead of reversing direction.
**Fix:** arrow-key handler now mirrors direction under `dir="rtl"` while preserving unchanged LTR
behavior for English.
**Test:** `apps/web/test/tabs-rtl-keyboard-nav.test.tsx`.

### 19. `apps/web/components/money/money-figure.tsx` — trend-arrow mirroring incorrect per tone
The directional trend arrow's RTL mirroring didn't correctly account for tone (at-risk vs. neutral
vs. recovered), producing a misleading direction cue in some combinations.
**Fix:** mirroring logic corrected per tone (at-risk and neutral mirror; recovered's `TrendingUp`
icon does not).
**Test:** `apps/web/test/money-figure-rtl-arrow-mirror.test.tsx`.

## Low

### 20. Focus-ring consistency sweep
Several interactive elements used a hand-rolled `focus-visible:ring-*` style instead of the shared
`.focus-ring` utility, producing a visually inconsistent keyboard-focus indicator: `CommandBar`
global search input, Overview page `ForwardCard` links, login demo-account buttons, `Button`, `Rail`
nav links, `AccountMenu` trigger.
**Fix:** all switched to the shared `.focus-ring` class.
**Test:** `apps/web/test/shell-focus-ring-consistency.test.tsx`,
`login-demo-button-focus-ring.test.tsx`.

### 21. `apps/web/components/modules/scrubber-table.tsx:54-58` — table row had a mouse-only click handler
`<TR>` carried `onClick`/`cursor-pointer` directly on the row with no keyboard equivalent.
**Fix:** removed the row-level `onClick`/`cursor-pointer` rather than bolting on
`role="button"`/`tabIndex` (which would have duplicated the row's existing focusable per-row
control and stripped native row/cell semantics an earlier fix in this same file already guards).
**Test:** `apps/web/test/scrubber-table.test.tsx`.

### 22. `Badge` "mock" variant used a non-AA money-neutral token
**Fix:** switched to `text-muted` (AA-contrast) instead of `text-money-neutral`.
**Test:** `apps/web/test/badge-mock-variant-contrast.test.tsx`.

## Considered / Not Independently Re-verified

- **`/ar/overview` live axe + keyboard re-check** — could not be completed cleanly this session.
  The chrome-devtools MCP session has a single shared "selected page" pointer; with 10+ sibling
  audit/verify/fix agents concurrently calling `select_page` on the same browser, the pointer kept
  getting stolen mid-check even with isolated-context tabs. Not a confirmed product bug — a tooling
  limitation. Recommend a clean single-agent re-check next time full browser exclusivity is
  available.
- **EN-dark header "Sign in" link + hero eyebrow contrast** — one axe run (via a scripted
  `classList.add('dark')`, not a real theme-toggle click) reported 2.94:1/3.35:1 against the near-
  black `--bg`. Could not be reproduced a second time via an actual click on the theme toggle button
  — likely a scripted-toggle artifact rather than a real bug (the eyebrow-line finding, #4 above, was
  independently confirmed and fixed separately). Flagged PLAUSIBLE, not CONFIRMED; worth a real
  click-through spot-check if revisited.
